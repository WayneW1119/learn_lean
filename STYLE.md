# STYLE.md — 风格约定

本文档记录写作风格、命名约定和工作偏好。可能随着项目推进而调整。

## 写作风格

- 中文写作，混合数学符号和 Python 风格伪代码
- 语法对象类型（如 `CoreTerm`、`Command`）用**代数数据类型**风格定义
  - 写出构造子、参数占位符、参数类型
  - 每个构造子附"写作"格式
- 状态对象类型（如 `State`、`Context`）用 `class / record` 风格定义
  - 字段通过 `S.field` 访问
- `PartialMap[K, V]` 表示从 K 到 V 的偏映射
- `isinstance(a, C2)` 用于模式匹配；`a.args` 拆解参数
- `写作 ...` 表示 term 的写法格式；`简记 X 为 ...` 是元语言层面的临时缩写
- 重要设计决策附**评论**，解释"为什么这样做"
- 函数定义用 Python 风格，附 docstring 和 `写作` 格式

## 命名约定

- `SurfaceTerm` 构造子以 `S` 为前缀（`SConst`、`SApp`、`SAnn` 等）
- `CoreTerm` 构造子无前缀（`Const`、`App`、`Proj` 等）
- 不引起歧义时，"写作"格式省略前缀
- Sigma 的字段名用 `a`/`b`（不是 Lean 的 `fst`/`snd`），因此投影名是 `Sigma.a`/`Sigma.b`
- 构造子参数用描述性名字（`structName`、`index`、`target` 等）
- binder 变量名用 `x`、`y`、`z`

## 版本原则

- **每个版本文档只关注与上一版本的差异**
- 只保留当前版本的变化标记（如 `【v10 新增】`、`【v10 修改】`），旧版本标记全部去掉
- 当前版本新增/修改的章节例子保留详细推导——这是读者理解该版本贡献的关键
- 非当前版本的例子只保留简要结论，去掉逐步推导
- 版本之间的关系是**累积**的——v10 包含 v0–v9 的所有内容，只是文档只标记变化

## 代码习惯

- 修改文件前先 Read 目标文件，不假设记得内容
- 用 `Edit` 工具做精确字符串替换；只有创建新文件或完全重写时才用 `Write`
- 匹配周围代码的风格：注释密度、命名、缩进、idiom
- 少用 shell 命令做文本处理（`cat`、`sed`、`grep`），优先用专门的 Read/Edit/Grep 工具

## 多参数应用缩写

`App` 是二元构造子，约定元语言缩写：

```
f a b c  意为  App(App(App(f, a), b), c)
```

例如 `(Const Sigma) A B` 意为 `App(App(Const Sigma, A), B)`。
